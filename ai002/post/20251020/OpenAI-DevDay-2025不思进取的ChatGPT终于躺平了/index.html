OpenAI DevDay 2025：不思进取的ChatGPT终于躺平了 --知识铺

2025年10月19日
| 阅读

当基座模型不再是护城河，OpenAI开始转向“应用生态”的构建。本文系统梳理 DevDay 2025 的发布内容，从 AgentKit、Codex 到 App SDK，揭示 ChatGPT 如何从模型能力转向平台能力，试图在“任务调度系统”与“开发者生态”中重塑自身定位。

国内AI卷大家是清楚的，飞书今天一个发布会，钉钉明天一定会跟进。但国外更是卷得不行，首先是Google基座模型（Gemini） + 图像视频套件（Nano Banana、Veo3）秀得飞起。

其次就是Meta也吃到了AI的巨大红利：

但从DeepSeek开源以来，LLama开源第一的位置坐得不稳，甚至后来还爆出了作假丑闻：

为解决技术瓶颈问题，这家伙几乎盯着OpenAI在挥锄头：

今年6月，Meta宣布组建超级智能实验室（Superintelligence Labs）。Meta计划斥资数十亿美元来吸引顶尖人才。目标是让该实验室拥有一支小规模但人才密度极高的团队。

综上，无论是Google的强势反超，还是Meta的高薪挖墙脚，或者国内DeepSeek、QWen等公司的强势追赶，无疑让曾经AI绝对的霸主感到心烦，于是OpenAI开始接连升级模型，只不过最近折腾的GPT-5并未让大家感到惊艳。

眼看着基座模型拉不开距离，OpenAI也就不演了，开始在应用侧发力，所以在10.7日凌晨OpenAI 的年度发布会，OpenAI Dev Day 2025开始了，整体看来，个人觉得可以用“不思进取”来形容……

按山姆·奥特曼的说法，这次发布会聚焦于如何让人更好的用 AI 去创造：App inside ChatGPT：“App Store”模式，吸引大量开发者入平台；Agent Kit：可以理解为字节体系的Coze全家桶；Codex 正式版：为了赶超Claude Code而提出的编程助手；多模态能力：gpt-image-1-mini（图像相关模型）、GPT-5 Pro、Sora、Real-Time Mini 的 API 都发布了；

可以看出，基座模型卷不明白，OpenAI开始做简单的事情了，比如排列组合做应用，其实上述的所有内容大家或多或少都看见过，并且没有哪一点是OpenAI特别擅长的，只不过OpenAI这次属于一个很好的技术路线发展整理者，更系统性的展示了这份蓝图。

操作系统

Apps inside ChatGPT，OpenAI推出自己的App SDK，其实这一说法出现得时间很早，比如之前红杉大会上就有：

云时代的 OS 是微软、移动时代是 iOS/Android、AI 时代的 OS，将不再是装机软件，而是任务调度系统。

什么是任务调动系统，最近大家最为熟悉的Manus以及正在进行变革的AI浏览器就是

然后模型本身能力不足，所以需要各种插件，于是MCP协议诞生，当前用得比较好的依旧是各种开发平台或者低代码开发平台如：Dify、Coze等。

另一方面，LLM 商店也不是今天才开始的，现在ChatGPT向开发者开放，允许他们在其中构建原生应用，其实逻辑与App Store、微信小程序是一样的，本质是一个生态游戏、平台流量游戏。

一般玩家希望平台给更多流量，平台希望接入更多开发玩家，从而增加平台能力，最后吸引更多的流量，在这个基础下OpenAI SDK应运而生。

了解其诞生契机后，OpenAI SDK提供的各种功能也就能够预料了，包括什么：数据链接、事件触发、互动卡片…

具体案例的话，也是挑选了几个典型的：在线课程、设计平台、房地产…

最后，有流量的地方就有GEO，出现GEO就会被各种控制，控制了的就不是好AI。平台化、商业化是不得不为，但如何维护一个良性的生态，避免灰产在其中予取予求，这也是OpenAI要提前考虑的。

AgentKit

AgentKit一套内置于 OpenAI 平台中的完整构建模块，旨在帮助开发者以更少的时间和精力，完成从原型设计到生产部署的全过程，他包含三大组件：

  1. Agent Builder (智能体构建器)：这是一个可视化的工作流设计工具，主要功能是拖拽；
  2. ChatKit (聊天工具包)：主要是一些UI语法糖和一些工作量减少的集成功能；
  3. Evals for Agents (智能体评估)：衡量智能体的性能，这个反而不知道要干撒；

说实话，AgentKit的出现是让我非常意外的，因为现在优秀的Agent平台已经太多了，Dify、Coze、n8n、FastGPT…

AgentKit无论怎么渲染，都摆脱不了一个事实：他是一个低代码平台，而低代码平台生存的前提有两点：

第一，他是对的、好用的；

第二，他有足够的流量分发给开发者。

所谓对的好用的主要看生态和投入度，如果OpenAI是想在这个领域持续发力，是战略的一部分，那么就要清晰其服务客服的用户画像，比如飞书、钉钉就很清晰，他们就是要争夺AI Office的流量。

但如果说AI Office是OpenAI的一部分，显然AgentKit的出发点就是错的

因为无论飞书还是钉钉其重点都是AI表格，从更本质的角度，比如企业视角来说，他们非常想要一套多人分散录入 → 集中汇总 → 统一分析 → 按权限查询的轻系统，这套轻系统就意味着AI办公的流量入口，一块非常大的蛋糕。

之前Excel/OA/低代码 都在抢这块份额，现在抢得尤其厉害的是飞书多维表格与钉钉AI表格。更确切的说法是，很多公司在挂着Coze、Dify的羊头在卖AI表格的狗肉。

只不过为什么AI表格是核心，因为在实践过程中实际使用部门非常唾弃Agent的体验，他们并不喜欢chat的体验，甚至有点排斥，他们已经习惯了Excel的操作，比如对HR、财务来说，他们能从一张Excel里面很容易就看到关键数据，而且这个动作很固定，Excel才是他们的工作台。

综上，如果OpenAI的目标是AI Office，那么他应该同步有强大的表格应用；如果不是，那么AgentKit这种低代码平台的意义是什么我就不清楚了，反正我身边的朋友要做生产级别智能体，一定会自己写代码…

最后，就算他们想玩这块，也没有自己的IM生态啊，所以AgentKit的出现非常神奇…

Codex

Codex的出现是最不意外的，因为我们之前就推衍过，无论是Coze还是多维表格，都是低代码平台的一部分，他们能提升效率的比例是有限的。

我们事实上需要的不是低代码平台，我们想要的是尽快进入自然语言编程时代。得益于GitHub的优秀语料，Cursor、Claude Code已经给了大家初步启示。

于是OpenAI再次玩起了什么都想要的老花样，愉快的“背刺”了其紧密合作的伙伴，核心功能完全建立在OpenAI模型之上的Cursor。

这可能会让Cursor感到崩溃，因为他过往的算法数据几乎都在提示词，而这些数据完全被OpenAI获得了，如果作为一个基座模型供应商站出来跟应用层公司抢饭碗，那将非常难看。

我之前是以为OpenAI可能会投资或者与Cursor战略合作去对抗Claude Code，结果他选择了自己玩，这也挺好的…

多模态

除此之外就是多模态技术相关的展示，特别需要说一嘴的就是最近火热且敏感的Sora2。

前些日子，Nano Banana爆了，国庆期间Sora App也爆了，说明在国内图片、视频类AI还是更受欢迎，因为这东西背后更见得着钱…

其实Sora2的爆发不能说是偶然，只能说他更舍得钱，因为在不久之前Google视频图像全家桶也做得非常不错。当时，谷歌发布了三款创作者向的 AI 工具：

  1. Flow 脚本→分镜→配乐→配音一条龙；
  2. Veo 3 让 AI 视频摆脱“无声时代”，支持原生音轨与物理细节；
  3. Imagen 4 图生图，2K 分辨率保持 Logo 与文字清晰；

这三者组合，就像是给创作者配齐了导演、摄影、视觉总监：

于是基友还在鼓励我去搞AI短剧：

但是，为什么Google套件没火起来呢，反而是后面的Nano Banana马上就火了呢？答案很简单：使用门槛和价格…

如果一个东西门槛高肯定是没准备好，可能是价格问题也可能是实现不完整的问题，于是5个月后OpenAI反而在这块拔的头筹了，当然也得益于效果确实逼真，只不过这个领先不会太久，2个月不能再多了…

但这里最大的问题可能还是监管和版权，视频这块维权比文字还恼火。

结语

近期，OpenAI在基座模型能力侧表现乏力，于是他们终于开始选简单的事情来做了，这让我想起了2年多之前的百度。

当时百度的文心大模型本身做得也没多好，其销售策略就已经展开了，各种在国内去做2B的生意（我们当时正好是医疗板块的选择之一）。

最终结果大家都看到了，基模做不好的团队2B怎么可能做得好？

这里就要劝老大哥OpenAI一下了：应用确实是AI的未来，但他是我们的未来，你们要不要专心做基座模型，应用的事情可以交给其他更擅长的公司去做，不要去消耗自己的精力，毕竟

Attention is all you need 的时代已经过去，现在颠覆可来得太快了，可能是几个月也可能是几周…
