Claude Code vs Codex，到底谁更强？ --知识铺

2025年10月8日
| 阅读

自从Codex问世，有关这个话题的争议就没停过

在9.29，Claude放出Sonnet 4.5后，这个话题吵的更厉害了…🤷

骁哥在公司高强度用了两天了（这句给老板看的），差不多心里有数了💁

直接放观点

模型选型：

  • Claude Code：Sonnet 4.5，这个没啥好说的

  • Codex：GPT-5-Codex-High，体感上和GPT-5-High差不多，只因为GPT-5-Codex毕竟是GPT-5在Codex的特供版

💁下文中，提到Claude-4.5其实就是指Claude Code，提到GPT-5就是指Codex。（应该没有在claudecode中用gpt的骚操作吧…）

Claude Code：主要的优势是速度。Claude 4.5的速度，是Codex的5-10倍😬

Codex：主要的优势是精度，Codex能注意到各种边缘细节，直戳要害；Claude 4.5就太喜欢发散了。

代码能力上，其实差不多。在有明确规划约束的前提下，其实Claude 4.5写代码还行。只要让Claude思考的越少，他就越强🐶

但如果说，就是要我从Claude Code和Codex中二选一，有你没我，有我没你…

那骁哥可能会选择Codex🥸

毕竟Codex擅长规划，能解决复杂场景问题，精度高很稳。慢就慢点吧，再慢也比人快🙆♀️

当然，分场景搭配使用，效果更佳👇

  • 追求速度用ClaudeCode；

  • 简单场景用ClaudeCode（比如给我添加一个小功能），复杂功能用Codex（比如重构）；

  • 新需求，先用ClaudeCode完成个大面，然后再用Codex一点点调Bug和重构；

  • 改Bug建议直接上Codex，当然也可以先用ClaudeCode尝试，两次不行换Codex；

或者，我们还可以玩的更花一点🤓

热血沸腾的组合技🔥

组合技一：Codex当ClaudeCode的主人

Codex出规划，ClaudeCode来执行规划，各取所长。这里就很适合用Spec Coding👉小白级Spec-kit实战教程，手把手教你Spec Coding，Codex负责到/task，然后ClaudeCode直接/implement

组合技二：互看不爽

一般是用ClaudeCode打先手，然后用Codex挑刺

比如我用ClaudeCode做完方案，或者写完代码，我就和Codex说：

[复制claudecode的方案]这是claudecode的方案，你怎么看  
看下git未提交的代码，这是claudecode刚写的，你怎么看

Codex就会开始挑刺👇

当然反过来也可以，只是Claude有讨好性人格，大概率会化身成Codex的舔狗…🐶

组合技三：各玩各的

核心思想是，让claudecode和codex完全分开同时干活，看最终谁效果好就留用谁

这个需要一点git的高阶技巧：git worktree👇

git worktree add ../app-claude -b feat/menu-claude  
git worktree add ../app-codex -b feat/menu-codex

执行完后，会多出两个工作目录

到目录下git branch一下，就显示这样

然后我们就可以一个起claudecode，一个起codex，同时开始干活。当然也是穿插使用一下挑刺法的。

最后根据结果，留一个你满意的，合进主干，另一个废弃。

小结一下

所以可以看出来，目前并没有啥完美的方案🤷

要精准，就要牺牲速度和money

要速度，就要牺牲准度

所以如果既要又要还要…那可能目前没有方案能满足你…

骁哥再举个例子，假腿，来Dota水友冒个泡🙋

如果你是一个高玩，追求极致👉那就切智利腿放技能；切敏捷腿喝魔瓶；切力量腿抗伤害

但如果说我就不切，懒得切，行不行，其实也行…

ClaudeCode，Codex，还是各个模型，其实也是一个道理。除非是追求极致，不然大可不必过度纠结～
